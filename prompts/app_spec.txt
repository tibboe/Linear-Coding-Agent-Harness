<project_specification>
  <project_name>GameOn - Football Matchmaking Platform</project_name>

  <overview>
    A mobile-first platform that aggregates player interest in football games at specific venues and times,
    then automatically creates games when sufficient interest exists. Players register their availability,
    skill level, and preferred venues. The platform shows real-time demand signals (how many players want
    to play at each venue/time). When thresholds are met, games are created and interested players are
    notified to join the lobby. Supports both open games (anyone can join) and private games (host approval
    required). Focus on football only for MVP to prove the concept.
  </overview>

  <core_concept>
    <interest_driven_matching>
      Traditional sports apps require someone to host a game first, then hope others join.
      This platform flips the model: players declare interest first, and games emerge organically
      when enough people want to play at the same time and place. This solves the "cold start"
      problem by showing demand before commitment.
    </interest_driven_matching>
    
    <demand_visualization>
      Players can see heatmaps of interest: "12 players want to play at Central Park on Saturday 2-4pm".
      This transparency creates social proof and helps players find games that will actually happen.
    </demand_visualization>
  </core_concept>

  <technology_stack>

      No authentication complexity initially - use demo accounts for testing.
      Mobile-first but web-based for Puppeteer testing compatibility.
      Clean separation between frontend and backend for future scaling.
    </philosophy>

      <port>3000</port>
    </backend>

    <database_choice_rationale>
      SQLite chosen for MVP because:
      - Zero configuration, single file database
      - No Docker or external services needed
      - Instant setup on any machine
      - Surprisingly powerful (handles millions of rows)
      - Easy to inspect with DB Browser for SQLite
      - Drizzle ORM makes migration to Postgres/Supabase trivial later
      - Perfect for solo development and Puppeteer testing
    </database_choice_rationale>

    <game_creation_system>
      <automatic_game_creation>
        When interest at a venue/time reaches the threshold:
        1. System detects threshold met
        2. Creates an "Open Game" automatically
        3. All interested players receive notification
        4. Players have X hours to confirm (join lobby)
        5. If minimum confirmed, game is locked in
        6. If not enough confirm, game cancelled, interests remain
      </automatic_game_creation>
